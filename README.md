# Provenance Guard

A Flask backend that estimates whether a piece of text was **written by a human**
or **generated by AI**, expresses the result as a plain-language transparency
label, records every decision in a structured audit log, and lets creators
contest a decision through an appeals workflow.

See [planning.md](planning.md) for the full system design.

## How it works

```
POST /submit → validate → rate limit → Signal 1 (Groq LLM)
                                     → Signal 2 (stylometric)
                                     → confidence scoring (0.65 / 0.35)
                                     → transparency label
                                     → audit log → JSON response
```

- **Signal 1 — Groq LLM** ([groq_detector.py](groq_detector.py)): `groq_classify(text) → {score, label}`.
- **Signal 2 — Stylometric** ([stylometric.py](stylometric.py)): `stylometric_analyze(text) → {ai_likeness_score}`.
- **Confidence engine** ([confidence.py](confidence.py)): `final_score = 0.65·groq + 0.35·stylometric`, banded as `<0.40 Human`, `0.40–0.69 Uncertain`, `≥0.70 AI`.
- **Transparency labels** ([labels.py](labels.py)): one of three fixed, plain-language labels per band.
- **Audit log** ([audit_log.py](audit_log.py)): append-only JSON Lines (`audit_log.jsonl`).

## Endpoints

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/submit` | Classify text; returns `content_id`, `attribution`, `confidence`, `label`. Rate-limited. |
| `POST` | `/appeal` | Contest a decision (`content_id` + `creator_reasoning`); flips status to `under_review`. |
| `GET`  | `/log` | Return recent audit-log entries (newest first; optional `?limit=N`). |

## Detection signals

The system combines **two independent signals** that look at different evidence.
The core design belief is that no single detector is trustworthy on its own, so
the value comes from having two signals that fail in *different* ways.

### Signal 1 — Groq LLM detector (`llama-3.3-70b-versatile`)

- **What it measures.** A holistic, semantic judgment: does the writing *read*
  like machine-generated text? The model weighs tone consistency, rhetorical
  structure, coherence, and the "fingerprint" of generated prose all at once,
  and returns a probability plus a label.
- **Why this signal.** LLMs are the only cheap way to capture *meaning-level*
  cues — e.g. that "leveraging synergistic solutions to unlock value" is hollow
  generated marketing copy. Surface statistics can't see that.
- **What it misses.** It is probabilistic and can be confidently wrong. It tends
  to over-flag **formal, polished human writing** (academic/legal prose) as AI,
  and under-flag **AI text a human has lightly edited**. It is also weakest on
  very short inputs, where there is little to judge.

### Signal 2 — Stylometric heuristics (pure Python, no external libraries)

- **What it measures.** Four objective, surface-level statistics of the text,
  each mapped to an "AI-likeness" contribution and averaged:
  1. **Sentence-length variation** — humans vary sentence length; AI is more uniform.
  2. **Vocabulary diversity** (type-token ratio) — humans use more distinctive vocabulary.
  3. **Repetition frequency** — AI repeats words/structures more.
  4. **Punctuation diversity** — humans use a wider variety of punctuation.
- **Why this signal.** It is transparent, deterministic, free, and — crucially —
  **independent of the LLM**. When it disagrees with Signal 1, that disagreement
  is itself informative and pulls the system toward "Uncertain."
- **What it misses.** It has no notion of *meaning*. It is fooled by
  vocabulary-rich fluent AI prose (reads it as human) and by formal human writing
  (can read it as AI). It is unstable on short text, where the TTR-based metrics
  saturate — so it deliberately returns a neutral `0.5` below two words.

**Why the two work together.** Signal 1 is the stronger *semantic* judge; Signal 2
is a cheap *surface* check that agrees on clearly-human text but can be inverted
by jargon. Because they draw on different evidence, agreement raises confidence
and disagreement lowers it — exactly the behavior we want from an ensemble.

## Confidence scoring

**How the signals are combined.** Both signals output an AI-likeness probability
in `[0, 1]`. They are merged with a fixed weighted average
([confidence.py](confidence.py)):

```
final_score = (0.65 × groq_score) + (0.35 × stylometric_score)
```

**Why this approach (and this weighting).** The LLM is the more reliable
*semantic* judge, so it carries the majority weight (0.65); the stylometric
signal is a lighter, independent corroborator (0.35). A weighted average — rather
than, say, "AI if either signal fires" — is what produces the desired
**conflict-resolution behavior**: when a confident LLM (say 0.85) disagrees with a
human-leaning stylometric score (0.25), the blend lands in the middle (~0.64) and
the system reports *Uncertain* instead of overcommitting. This encodes the
project's core bias: **favor uncertainty over falsely accusing a human creator.**

**Thresholds** (the label categories, from [planning.md](planning.md) §4):

| `final_score` | Classification |
| --- | --- |
| `< 0.40` | Likely Human |
| `0.40 – 0.69` | Uncertain |
| `≥ 0.70` | Likely AI |

**How I validated that scores are meaningful (not constant).** I tested inputs
chosen to sit in different bands and confirmed the combined score tracks them,
and I swept the classifier across all values `0.000–1.000` to confirm it
reproduces the planning thresholds exactly (no drifted boundaries). Two
representative cases from that testing, showing **noticeably different**
confidence and both signal scores:

| Case | Signal 1 (Groq) | Signal 2 (stylometric) | Combined | Label |
| --- | --- | --- | --- | --- |
| **Higher-confidence** — repetitive, uniform AI text | 0.90 | 0.63 | **0.805** | Likely AI-generated |
| **Lower-confidence** — formal academic human prose | 0.80 | 0.49 | **0.698** | Uncertain |

The first is a clear, high-confidence AI call; the second is a genuinely
ambiguous case where the LLM leans AI but the blend stays in the *Uncertain*
band — a ~0.11 spread across two deliberately different inputs, confirming the
score varies with the content rather than returning a constant.

**What I'd change deploying this for real.**
- **Calibrate against a labeled dataset.** The 0.65/0.35 weights and the
  0.40/0.70 thresholds are reasoned defaults, not empirically fit. With real
  labeled human/AI data I'd tune them to a target false-positive rate on human
  creators.
- **Stop treating raw LLM output as calibrated probability.** I'd apply
  probability calibration (e.g. isotonic/Platt scaling) so `0.7` means the same
  thing across input types.
- **Add signal-specific confidence.** Both signals degrade on short text; I'd
  down-weight a signal when its own input is too short to be reliable rather than
  averaging it in at full weight.

## Transparency labels

The label text is fixed per band (verbatim, no paraphrasing) and generated by
`generate_label(confidence)` in [labels.py](labels.py). All three variants:

| Band (`confidence`) | Classification | Exact label text |
| --- | --- | --- |
| `≥ 0.70` | High-confidence AI | `Likely AI-generated content.`<br>`Our analysis indicates strong evidence that this text was generated using AI systems. This classification is made with high confidence based on multiple signals.` |
| `< 0.40` | High-confidence Human | `Likely human-written content.`<br>`Our analysis indicates strong evidence that this text was written by a human creator. This classification is made with high confidence based on multiple signals.` |
| `0.40 – 0.69` | Uncertain | `Uncertain attribution.`<br>`We found mixed or inconclusive signals in this text. We cannot confidently determine whether it was written by a human or generated by AI. Results are intentionally conservative to avoid misclassification.` |

Each label is a two-line string (headline `\n` explanation). The three variants
were confirmed reachable end-to-end by submitting inputs in each band.

## Appeals workflow

A creator who disagrees with a decision submits an appeal with their reasoning.
No automatic re-classification happens — the appeal is recorded and the status
changes to `under_review`, preserving the original decision for human review.

Request:

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": 3, "creator_reasoning": "This is my own writing; I tend to be repetitive when explaining technical ideas."}' | python -m json.tool
```

Response:

```json
{
  "message": "Appeal submitted successfully.",
  "content_id": 3,
  "status": "under_review"
}
```

The appeal is then visible in the audit log with `"status": "under_review"` and
the creator's reasoning (see the appeal entry in the [Audit log](#audit-log)
sample below). The original submission entry is left unchanged, so the original
classification and confidence are preserved.

## Audit log

Every decision is recorded in an append-only **JSON Lines** file
(`audit_log.jsonl`) — structured, one JSON object per line, never mutated.
`GET /log` returns these entries as JSON.

**Submission entries** capture: `timestamp`, `content_id`, `attribution`
(the combined classification), `confidence` (combined score), and **both
individual signal scores** (`llm_score` = Signal 1, `stylometric_score` =
Signal 2), plus `creator_id` and `status`.

**Appeal entries** (`"type": "appeal"`) record that an appeal has been filed
for a `content_id`, snapshotting the `original_classification` and `confidence`
and setting `status` to `under_review` — the original submission row is left
untouched, so the original decision is preserved. Whether a submission has been
appealed is captured by the presence of a linked appeal entry.

Sample log (3 submissions spanning all bands + 1 appeal); each entry shows the
attribution result, confidence score, and timestamp:

```json
[
  { "content_id": 1, "creator_id": "nadia", "timestamp": "2026-07-01T17:37:10.842Z",
    "attribution": "Human", "confidence": 0.2603, "llm_score": 0.23,
    "stylometric_score": 0.3167, "status": "classified" },
  { "content_id": 2, "creator_id": "maria", "timestamp": "2026-07-01T17:37:12.276Z",
    "attribution": "Uncertain", "confidence": 0.6977, "llm_score": 0.8,
    "stylometric_score": 0.5077, "status": "classified" },
  { "content_id": 3, "creator_id": "bot-x", "timestamp": "2026-07-01T17:37:12.992Z",
    "attribution": "AI", "confidence": 0.8025, "llm_score": 0.9,
    "stylometric_score": 0.6215, "status": "classified" },
  { "type": "appeal", "content_id": 3, "original_classification": "AI",
```

Inspect it live with `curl -s http://localhost:5000/log | python -m json.tool`.

## Rate limiting

The submission endpoint is protected with
[Flask-Limiter](https://flask-limiter.readthedocs.io/), keyed on client IP
(in-memory storage for local dev):

```python
@app.post("/submit")
@limiter.limit("10 per minute;100 per day")
```

### Chosen limits and reasoning

| Limit | Purpose |
| --- | --- |
| **10 per minute** | Absorbs realistic bursts — a writer submitting a piece, tweaking it, and resubmitting a few times, or checking a handful of drafts back-to-back. A human editing their own work almost never exceeds ~10 submissions in a single minute, but an automated script trying to flood the API (or brute-force the classifier) hits this cap almost immediately. It also caps cost/abuse of the billable Groq call — the limit is enforced *before* the model is invoked. |
| **100 per day** | Bounds sustained usage over a full day. Even a prolific creator publishing and re-checking many pieces rarely approaches 100 submissions/day, so genuine users are unaffected, while a script that paces itself under the per-minute limit still can't grind against the system indefinitely. |

The two limits work together: the **per-minute** cap stops rapid flooding, and
the **per-day** cap stops slow, sustained abuse that would otherwise stay under
the minute limit. Both are deliberately set well above normal single-author
usage so legitimate writers never see a `429`, and well below the volume a
flooding script would generate.

Requests exceeding a limit receive HTTP **429 Too Many Requests**.

> **Production note:** in-memory storage is per-process. A multi-worker
> deployment would set `storage_uri` to a shared backend (e.g. Redis) so limits
> are enforced across all workers.

## Known limitations

**Polished, vocabulary-rich AI essays are systematically under-scored and tend
to land in *Uncertain* rather than *AI*.** This is not a generic disclaimer — it
is a direct, measured consequence of Signal 2's design. The stylometric signal's
two vocabulary-based metrics (type-token ratio and repetition) *saturate toward
"human"* on short, fluent, well-written passages, because such text rarely
repeats words — so TTR ≈ 0.9 regardless of the actual author. On a clean AI essay
I measured the stylometric score at only ~0.30 even while the LLM was confidently
0.80. Because stylometry is weighted 0.35, it drags the blend down: with the
stylometric score stuck near 0.30, the AI band (`≥ 0.70`) effectively requires the
LLM to be ≥ 0.92 (`0.65·g + 0.35·0.30 ≥ 0.70`). So a genuinely AI-written,
grammatically clean essay will often be reported as *Uncertain*.

This fails in the "safe" direction (it never *falsely* labels such text as
human), which is consistent with the project's bias — but it is a real
sensitivity ceiling. A related weakness from the same root cause is **very short
text** (tweets, one-line poetry like `"rain. rain. rain."`): stylometry has too
few tokens to be stable and returns a neutral `0.5`, so those cases lean almost
entirely on the LLM.

The fix (see [Confidence scoring](#confidence-scoring) → "what I'd change") is
length-aware weighting and empirically calibrated thresholds.

## Spec reflection

**One way the spec guided the implementation.** The planning doc pinned down the
confidence bands (`<0.40 / 0.40–0.69 / ≥0.70`) and the exact transparency-label
text *before* any code existed. That contract kept the confidence engine, the
label generator, and the audit log all consistent, and it gave me an objective
test to write: I swept every score `0.000–1.000` and asserted the classifier
reproduced the planned bands exactly. Without a fixed spec I'd likely have let the
boundaries drift ("0.7 feels better than 0.65"); having it written down turned a
judgment call into a verifiable check.

**One way the implementation diverged from the spec, and why.** The planning doc
(§4.3) modeled the audit log as a **single record per submission that is mutated
in place** on appeal — the `GET /log` example shows one row whose `status`
changes to `under review` and that gains an `appeal` field. I implemented the log
as **append-only** instead: an appeal is written as a *separate* linked entry and
the original submission row is never touched. The reason is audit-log integrity —
an audit trail whose past entries can be rewritten isn't trustworthy as evidence,
and preserving the original decision verbatim is exactly what an appeals process
should do. The trade-off is that "current status" is now derived from the latest
entry for a `content_id` rather than read from one mutable field. (I also renamed
the request fields from the planning doc's `author`/`content` to `creator_id`/
`text` to match the platform's field convention.)

## AI usage

This project was built with an AI coding assistant. Two concrete instances,
including what I overrode:

**1. Generating Signal 2 + the confidence engine, then overriding a "fix."**
I directed the AI to implement `stylometric_analyze` and `compute_confidence`
under a strict spec (four named metrics, the 0.65/0.35 formula, the fixed
thresholds, no external libraries). It produced working, correctly-weighted
functions that passed my threshold sweep. But when I then tested four labeled
inputs, a clearly-AI essay only reached *Uncertain*. I had the AI break the
stylometric score into its four sub-metrics, which showed the TTR/repetition
metrics were saturating toward "human" on short fluent text. The AI offered to
recalibrate the signal or shift the weights — **I overrode that and kept the
scoring unchanged**, because the miss fails safe (toward uncertainty, never
falsely accusing a human) and matches the documented design bias. Instead of a
silent code change, that decision became the [Known limitations](#known-limitations)
section.

**2. Designing the audit log / appeals storage, and diverging from the plan.**
I directed the AI to add the audit log and the `POST /appeal` workflow. Its first
instinct followed the planning doc literally — mutate the submission record in
place to flip its status on appeal. **I decided against that and had it implement
an append-only log** with appeals as separate linked entries (see
[Spec reflection](#spec-reflection)), because a mutable audit trail defeats the
purpose of an audit trail. I also had it add a defensive detail I asked for
specifically: the appeal lookup matches `content_id` whether it arrives as an int
or a string, and invalid/unknown IDs return `400`/`404` rather than silently
succeeding.
